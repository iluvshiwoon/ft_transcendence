# 4thewin

A multiplayer Connect 4 platform with real-time gameplay, AI opponents, user profiles, friends system, chat, and stats tracking. Built with Astro + React, Fastify + TypeScript, PostgreSQL, and deployed as containers behind a WAF.

## Description

**4thewin** is a web-based Connect 4 game where players can:

- Play against an AI opponent with three difficulty levels (easy, medium, hard)
- Challenge other registered players in real-time 1v1 matches
- Create public or private lobbies with custom game settings
- Track their game statistics, win rate, and ELO rating
- Chat directly with other users
- Manage a friends list and see who is online

The project is a full-stack application composed of 7 containerized services orchestrated by a single command. The frontend is responsive and accessible across desktop, tablet, and mobile viewports, built with Tailwind CSS utility classes. Compatible with the latest stable Google Chrome.

## Instructions

### Prerequisites

- **Container runtime**: `podman` + `podman-compose` (Linux), or Docker (macOS via OrbStack / Docker Desktop). The Makefile auto-detects which to use.
- **pnpm** — for the local dev workflow (`make dev`). Install: `brew install pnpm` or `npm i -g pnpm`.
- **`.env`** at repo root (copy from `.env.example`). The `.env` file is excluded from version control via `.gitignore` — no credentials are committed to the repository.

### First-time setup

```sh
cp .env.example .env
make build
```

This builds all images and starts the full stack. The HTTPS endpoint uses a self-signed certificate (auto-generated by the OWASP ModSecurity image). Chrome will show a "Not Secure" warning — click "Advanced" → "Proceed to localhost" during the demo.

### Development

```sh
make dev    # backend in compose + Astro dev server at localhost:4321
```

### Useful commands

```sh
make rebuild-server       # after backend changes (~10s)
make rebuild-web          # after frontend changes (~15s)
make logs-server          # tail backend logs
make down                 # stop everything (state preserved)
make clean                # stop + remove volumes (nukes DB + Vault)
make help                 # lists every target
```

## Architecture

```
Browser ──HTTPS──> ModSecurity (WAF, :8443) ──> Astro SSR (web, :4321) ──> Fastify API (server, :3000) ──> PostgreSQL (:5432)
                                                                       │
                                                                  Socket.io
                                                                       │
                                                              HashiCorp Vault (:8200)
```

- **Request flow**: All external traffic enters via HTTPS at the WAF reverse proxy. The WAF applies ModSecurity rules (OWASP CRS, paranoia level 2) before forwarding to the Astro frontend. API requests are proxied to the Fastify backend.
- **Real-time**: Socket.io handles game events, chat messages, and notification delivery over WebSocket connections.
- **Secrets**: Vault stores JWT_SECRET, DATABASE_PASSWORD, and OAUTH42 credentials. The server reads them at startup via HTTP API. Falls back to environment variables when Vault is unavailable (local dev).
- **Container orchestration**: 7 services (postgres, vault_server, vault_init, vault_unseal, server, web, mod_security) defined in `compose.yml`. Single `make build` command brings everything up.

## Known Limitations

- **Connect 5**: The database schema supports a `connect5` mode, but the win detection is hardcoded to 4-in-a-row. The mode selector is available in lobbies but does not change the win condition.
- **HTTPS certificate**: Uses a self-signed certificate. Browsers will show a security warning.
- **OAuth**: Requires a registered 42 OAuth application. Credentials must be pushed into Vault manually on first setup.
